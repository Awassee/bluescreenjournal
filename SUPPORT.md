# Support

## Where to start

If you need help using bsj, start with:

1. `README.md`
2. `docs/QUICKSTART.md`
3. `docs/FAQ.md`
4. `bsj settings`
5. `bsj doctor`
6. `bsj doctor --unlock`

## When to open a GitHub issue

Open an issue if you can reproduce a bug, packaging problem, installer failure, documentation gap, or product UX problem.

Please include:

- macOS version
- Terminal.app or iTerm2 version
- how bsj was installed
- exact command run
- expected behavior
- actual behavior
- output from `bsj settings`
- output from `bsj doctor` and `bsj doctor --unlock` when safe

Do not paste secrets or journal content.

## Feature requests

Open a feature request when the product is missing a clear workflow, not just when a preference differs from the current design.

Good requests explain:

- the user problem
- the current workaround
- the value of solving it
- why the change fits the product direction

## Security issues

Do not open public issues for suspected security vulnerabilities.

Use `SECURITY.md`.
