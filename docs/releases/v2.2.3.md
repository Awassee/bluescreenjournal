# v2.2.3

This is a focused installer reliability patch.

It does not change the writing model. It fixes the exact post-install launch path that could still fail for first-run users when the installer was started via `curl | bash`.

## Highlights

- fixed installer menu option `1` (`Launch BlueScreen Journal here now`) for piped installer sessions
  - the same-terminal handoff now runs the TUI under a clean PTY
  - this prevents `Failed to initialize input reader` at the moment of first launch
- added real TUI launch smoke coverage for installer validation
  - packaged installer smoke now tests the piped installer + option `1` path
  - public installer smoke now tests the same launch-here path too
- kept the patch intentionally narrow
  - no journaling workflow or vault behavior changes
  - the goal is a more trustworthy stable release

## Why this release exists

`v2.2.2` improved installer clarity, but one edge remained:

- install could succeed
- the post-install menu could appear correctly
- then the launch-here action could fail because the TUI was not getting a clean terminal input context

That is exactly the kind of first-run bug that undermines trust.

`v2.2.3` closes that gap and adds a regression test for the real launch path instead of a shallow CLI-only check.

## Validation

- `cargo fmt --all`
- `cargo test --all-targets`
- `./scripts/package-release.sh --universal`
- piped installer launch-here regression log now reaches `BSJ_TUI_SMOKE_OK`

Release QA note:

- full installer matrix later hit local disk pressure in the source-update smoke branch on the maintainer machine
- the launch-here regression itself passed before that environment-only failure

## Install

Latest stable:

```bash
curl -fsSL https://raw.githubusercontent.com/Awassee/bluescreenjournal/main/install.sh | bash
```

Pin this release:

```bash
curl -fsSL https://raw.githubusercontent.com/Awassee/bluescreenjournal/main/install.sh | bash -s -- --version v2.2.3
```

Debug installer state on a problem machine:

```bash

```
